# 實驗室 fMRIPrep 運算環境架構與資料規範說明會議

**會議目的：** 
與大家同步實驗室伺服器在執行 fMRIPrep 神經影像前處理流程時的建議作業模式，期望在確保系統底層安全性的同時，也能協助大家建立統一的資料格式，讓整體的運算流程更加順暢。

---

## 1. 容器化環境之安全考量與政策調整

過去大家可能習慣使用 `fmriprep-docker` 作為執行介面。但在實驗室多用戶共享的伺服器架構下，這種作法會帶來一些我們需要共同避免的系統風險。

*   **風險說明：** Docker Daemon 在 Linux 系統底層預設是以 `root` 權限持續運行的。若開放大家使用 `fmriprep-docker`，系統在架構上必須給予所有使用者較高的管理員權限（如加入 docker 群組）。這較不符合資訊安全的「最小權限原則」，若操作不慎，可能會影響到伺服器的整體穩定性與其他同學的運算作業。
*   **業界規範參考：** 為了系統安全，國際大型運算中心通常會建議採用更安全的替代方案。正如 fMRIPrep 官方文件 (v0.5.1) 所述：
    > *"For security reasons, many HPCs (e.g., TACC) do not allow Docker containers, but do allow Singularity containers."*

基於上述維運與安全考量，我們會建議大家在伺服器上避免透過 Docker 執行 fMRIPrep，並逐步轉移至更適合共享環境的工具。

---

## 2. 推薦的替代方案：Apptainer (原 Singularity)

為了解決大家的運算需求並兼顧伺服器安全，實驗室推薦大家使用 **Apptainer**（前身為 Singularity）作為標準的容器執行環境。

### Apptainer 的架構優勢：
1.  **無特權執行 (Rootless Execution)：** 這是專為高效能運算 (HPC) 設計的技術。它不需要背景的 root daemon 即可運行，安全性極高。
2.  **使用者權限繼承：** 系統會自動對應你原本的使用者身份，確保大家只能存取自己擁有權限的資料夾，避免了誤觸系統核心檔案的風險。
3.  **高整合度與便利性：** 映像檔會被打包成單一的 `.sif` 檔案，不僅方便在不同電腦間移動，預設也會自動掛載當前目錄，讓路徑設定變得相對簡單。

**後續行動：** 實驗室稍後會提供 Apptainer 版本的 fMRIPrep 執行指令範本，讓大家撥空熟悉其基本操作。

---

## 3. 運算前置作業：BIDS 資料標準化建議

fMRIPrep 是一個高度自動化的處理管線，為了讓它能順利讀取大家的資料，輸入的檔案結構需要盡可能符合 **BIDS (Brain Imaging Data Structure)** 規範。若結構或命名不一致，可能會導致運算中斷，影響大家的研究進度。

### 標準驗證流程建議：
為了節省大家除錯的時間，我們強烈建議在將資料放上伺服器運算之前，先透過官方的 BIDS Validator 進行自我檢核。

*   **工具連結：** [BIDS Validator](https://bids.neuroimaging.io/tools/validator.html)
*   **使用說明：** 
    1. 這是一個非常方便的網頁端檢核工具。
    2. 請直接在瀏覽器中選擇你本機端整理好的資料夾。
    3. **隱私與安全提示：** 此工具的設計十分安全，它僅會在你的本地端瀏覽器解析檔案的「檔名」與「目錄結構」，**絕對不會**將你的醫學影像檔案上傳至任何雲端伺服器，因此請安心使用。
    4. 建議大家在確認網頁顯示 "0 Errors" 後，再將資料上傳至實驗室伺服器，這能確保你的 fMRIPrep 運算一次就能順利跑完。
